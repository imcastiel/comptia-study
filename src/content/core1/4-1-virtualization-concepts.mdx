# Virtualization Concepts

## What Is Virtualization?

Virtualization is the process of creating a software-based (virtual) representation of physical hardware — including CPUs, RAM, storage, and networking — allowing multiple isolated virtual environments to run on a single physical host.

### Why Virtualization Is Used

| Benefit | Description |
|---|---|
| Server consolidation | Run many virtual servers on one physical host, reducing hardware costs |
| Isolation | Each VM is independent — a crash or compromise in one VM does not affect others |
| Snapshots | Save the VM's state at a point in time for easy rollback |
| Test environments | Spin up and destroy environments without physical hardware |
| Disaster recovery | VMs can be backed up as files and restored on any compatible host |
| Legacy app support | Run old OSes in a VM on modern hardware |
| Fast provisioning | Deploy a new server in minutes by copying a VM image |

<ConceptCallout type="info">
The CompTIA A+ exam focuses on the practical concepts and common configurations of virtualization, not deep hypervisor administration. Know the types, the terminology, the networking modes, and the hardware requirements.
</ConceptCallout>

## Hypervisor Types

A **hypervisor** (also called a Virtual Machine Monitor or VMM) is the software that creates and manages virtual machines.

### Type 1 — Bare-Metal Hypervisor

A Type 1 hypervisor runs **directly on the physical hardware** with no host operating system underneath it. It has direct access to the hardware and manages all hardware resources.

| Hypervisor | Vendor | Notes |
|---|---|---|
| VMware ESXi | VMware (Broadcom) | Industry-leading enterprise hypervisor |
| Microsoft Hyper-V | Microsoft | Built into Windows Server; also available in Windows 10/11 Pro |
| KVM (Kernel-based VM) | Open source (Linux) | Integrated into the Linux kernel |
| Xen | Open source / Citrix | Used by Amazon AWS (historically) |
| Oracle VM Server | Oracle | Enterprise use |

**Characteristics:**
- Highest performance (no host OS overhead)
- Used in enterprise data centers
- Requires dedicated hardware — you don't use it as a daily-use computer
- Managed via web interface or management console (e.g., vCenter for ESXi)

### Type 2 — Hosted Hypervisor

A Type 2 hypervisor runs **as an application on top of a host operating system** (Windows, macOS, Linux). The host OS controls the hardware and the hypervisor accesses hardware through the host OS.

| Hypervisor | Vendor | Host OS | Notes |
|---|---|---|---|
| VMware Workstation | VMware | Windows/Linux | Professional desktop virtualization |
| VMware Fusion | VMware | macOS | Popular for Mac users running Windows VMs |
| Oracle VirtualBox | Oracle | Windows/macOS/Linux | Free, open source |
| Parallels Desktop | Parallels | macOS | Best macOS-to-Windows performance |
| QEMU | Open source | Linux/macOS/Windows | Often paired with KVM |

**Characteristics:**
- Runs alongside other applications
- Easier to set up; suitable for development, testing, and learning
- Slight performance overhead from the host OS layer
- You can use the host OS for regular work while VMs run in the background

<ConceptCallout type="tip">
Type 1 vs Type 2 is a core exam question. The mnemonic: **Type 1 = on the metal** (no OS between it and hardware). **Type 2 = on top of an OS** (requires an existing operating system). Microsoft Hyper-V is technically a Type 1 hypervisor even though it installs inside Windows — when Hyper-V is enabled, Windows itself runs in a partition managed by the Hyper-V hypervisor.
</ConceptCallout>

## Virtual Machine Components

### Virtual CPU (vCPU)

A vCPU is a logical representation of a CPU core assigned to a virtual machine. The hypervisor schedules vCPU operations on the physical CPU cores.

- Overcommitment is possible (more vCPUs assigned than physical cores) — the hypervisor time-slices them
- More vCPUs does not always equal more performance — it depends on scheduling contention
- Modern hypervisors support NUMA-aware scheduling to minimize cross-socket latency

### Virtual RAM

RAM is allocated from the host's physical RAM to each VM. Unlike CPUs, RAM generally cannot be significantly overcommitted without performance degradation (though features like memory ballooning and transparent page sharing help).

- Each running VM consumes its allocated RAM from the host's pool
- Over-allocating RAM causes the host to swap VM memory to disk — catastrophic for performance

### Virtual NIC (vNIC)

Each VM gets one or more virtual network adapters. The hypervisor connects vNICs to virtual switches, which connect to the physical network adapters on the host.

- VMs appear to the network as independent hosts with their own MAC addresses
- Multiple vNICs can be assigned to one VM (useful for firewall or router VMs)

### Virtual Disk Formats

| Format | Extension | Platform | Notes |
|---|---|---|---|
| VMDK | .vmdk | VMware | VMware's native format; widely supported |
| VHD | .vhd | Microsoft | Hyper-V and older Virtual PC |
| VHDX | .vhdx | Microsoft | Improved VHD; supports up to 64 TB |
| QCOW2 | .qcow2 | QEMU/KVM | Supports snapshots and compression |
| OVA/OVF | .ova/.ovf | Cross-platform | Open Virtualization Format; portable |

<ConceptCallout type="info">
**Thick vs Thin Provisioning:**
- **Thick-provisioned disk:** All storage space is allocated upfront — a 100 GB disk takes 100 GB on the host immediately. Better for predictable performance.
- **Thin-provisioned disk:** Storage space is allocated on demand as data is written. A 100 GB thin disk might only consume 10 GB initially. Better for storage efficiency but can cause problems if the storage pool fills up.
</ConceptCallout>

### Virtual Switch (vSwitch)

A virtual switch operates like a physical Layer 2 network switch but entirely in software within the hypervisor. It connects VMs to each other and to the physical network.

- VMs on the same vSwitch and same network segment can communicate without traffic leaving the host
- Port groups on a vSwitch can be assigned different VLANs

## VM Networking Modes

### NAT (Network Address Translation)

The VM shares the host's IP address. The hypervisor performs NAT between the VM and the external network.

- **VM can reach the internet** through the host
- **External network cannot initiate connections** to the VM (without port forwarding)
- VM gets a private IP from the hypervisor's built-in DHCP server
- Easiest mode for internet access without network configuration

### Bridged

The VM is connected directly to the physical network as if it were a separate physical machine. It gets its own IP address from the network's DHCP server (or must be configured with a static IP).

- **VM appears as a separate host** on the physical network
- Other devices on the network can see and connect to the VM
- Requires a physical network or access point that allows it
- Used when the VM needs to be accessible to other network devices (e.g., running a test web server)

### Host-Only

The VM can communicate **only with the host** and with other VMs on the same host-only network. No external network access.

- Completely isolated from the physical network and internet
- A separate virtual network adapter is created on the host for this purpose
- Used for isolated test environments and lab setups
- Perfect for practicing network configuration without risk

### Internal Network (VirtualBox) / Private vSwitch (Hyper-V)

VMs can communicate with each other but not with the host or external network. Even more isolated than host-only.

<ConceptCallout type="tip">
Networking mode exam cheat sheet:
- **NAT:** VM can reach internet, internet cannot reach VM, simplest setup
- **Bridged:** VM is on the same network as physical devices, gets its own IP
- **Host-Only:** VM talks only to host and other VMs, no internet
- **Internal:** VM talks only to other VMs on the same internal network, not even the host
</ConceptCallout>

## Snapshots

A **snapshot** captures the complete state of a virtual machine at a specific point in time, including:
- The contents of RAM (memory state)
- The state of all virtual disks
- VM configuration (number of vCPUs, RAM allocation, settings)

### When to Use Snapshots

- **Before applying system updates** — rollback if something breaks
- **Before testing software installations** — revert if the software causes problems
- **Before making configuration changes** — quick undo capability
- **Creating a clean baseline** — snapshot a fresh OS install for quick provisioning

### Snapshot Behavior

When a snapshot is taken, the current disk becomes read-only. A **delta disk** (differencing disk) records all subsequent changes. The snapshot chain tracks changes from the base:

```
Base Disk (read-only after first snapshot)
    └── Snapshot 1 Delta (changes since base)
            └── Snapshot 2 Delta (changes since Snapshot 1)
```

### Rollback / Revert

Reverting to a snapshot discards all changes made after the snapshot and restores the VM to that exact state. This is nearly instant.

<ConceptCallout type="warning">
**Snapshots are not backups.** They are stored on the same storage as the VM. If the storage fails, both the VM and all its snapshots are lost. For true backup, export the VM to a separate location. Also, long snapshot chains degrade performance and consume significant additional storage — clean up old snapshots regularly.
</ConceptCallout>

## Hardware Requirements for Virtualization

### CPU Virtualization Extensions

Virtualization requires hardware support in the CPU:
- **Intel:** Intel VT-x (Virtualization Technology for IA-32/64)
- **AMD:** AMD-V (AMD Virtualization) / SVM (Secure Virtual Machine)

These must be **enabled in BIOS/UEFI settings**. On many systems they are disabled by default. The setting is usually found under:
- BIOS → Advanced → CPU Configuration → Intel Virtualization Technology → Enable
- BIOS → Advanced → SVM Mode → Enable

**Secondary extensions:**
- **Intel VT-d / AMD IOMMU:** Required for PCI passthrough (assigning a physical PCIe device directly to a VM)
- **Intel EPT / AMD NPT (RVI):** Extended/Nested Page Tables — hardware-accelerated memory virtualization for better performance

<ConceptCallout type="warning">
If you try to run a 64-bit VM and get an error like "VT-x is not available" or "AMD-V is disabled," the CPU virtualization extension is disabled in BIOS/UEFI. Reboot, enter setup (usually Del or F2), find the virtualization setting, enable it, save and exit.
</ConceptCallout>

### Resource Allocation Guidelines

| Resource | Consideration |
|---|---|
| CPU | Assign 1–2 vCPUs for basic VMs; don't over-allocate |
| RAM | Allocate enough for the guest OS plus workload; reserve some for the host |
| Storage | Thin-provision to save space; ensure host has adequate free space |
| Network | 1 vNIC is usually sufficient; bridged for network visibility |

## VDI — Virtual Desktop Infrastructure

**VDI** is a deployment model where user desktops run as virtual machines on centralized servers in a data center. Users connect to their desktop VM remotely using thin clients, laptops, or other devices.

| Component | Role |
|---|---|
| VDI server | Runs many desktop VMs (Type 1 hypervisor) |
| Connection broker | Routes users to their assigned or available desktop VM |
| Thin client | Lightweight device used to connect to VDI session |
| Display protocol | RDP, ICA/HDX (Citrix), PCoIP (VMware Horizon) |

**Benefits:**
- Centralized management — patch and update all desktops from one place
- Data stays in the data center (not on endpoints)
- Users can access their desktop from anywhere
- Easy provisioning/deprovisioning for employees

**Drawbacks:**
- Requires reliable low-latency network connection
- Higher server hardware costs
- Graphics-intensive applications may not perform well (unless GPUs are virtualized)

## Containers vs VMs

### Virtual Machines

- Include a full guest OS (kernel + userspace)
- Isolated at the hardware level via hypervisor
- Heavier resource footprint
- Boot time: seconds to minutes
- Strong isolation — each VM has its own kernel

### Containers

- Share the host OS kernel
- Isolated at the OS process level
- Very lightweight (no full OS)
- Start in milliseconds
- Less isolated — a kernel exploit could affect all containers

| Feature | Virtual Machine | Container (e.g., Docker) |
|---|---|---|
| OS | Full guest OS | Shares host kernel |
| Isolation | Hardware (hypervisor) | OS namespace/cgroup |
| Size | GBs | MBs |
| Startup time | Seconds–minutes | Milliseconds |
| Overhead | High | Low |
| Use case | OS-level isolation, legacy apps | Microservices, rapid deployment |

<ConceptCallout type="info">
**Docker** is the most common container platform. A Docker image packages an application with its dependencies but shares the host Linux kernel. Docker containers run on Linux natively; on Windows/macOS, Docker runs inside a lightweight Linux VM. For the A+ exam, understand that containers are lighter than VMs because they don't include their own OS.
</ConceptCallout>

## Emulators vs Hypervisors

| Aspect | Emulator | Hypervisor |
|---|---|---|
| What it simulates | Different hardware architecture | Same hardware architecture |
| CPU translation | Software interprets every instruction | Hardware extensions run native code |
| Performance | Much slower (full translation overhead) | Near-native speed |
| Use case | Running ARM apps on x86 (e.g., Android emulator on PC), retro gaming | Running the same OS/apps on virtual hardware |
| Example | QEMU (without KVM), Android Studio emulator, game console emulators | VMware, Hyper-V, VirtualBox |

<ConceptCallout type="tip">
An **emulator** pretends to be completely different hardware (like an x86 PC emulating an ARM processor). This requires software translation of every instruction — very slow. A **hypervisor** runs the guest on the same CPU architecture as the host, using hardware extensions for efficiency. The Android Emulator in Android Studio actually uses a hypervisor (HAXM, Hyper-V, or KVM) when running x86 Android images for speed, and only falls back to full emulation for ARM images.
</ConceptCallout>
