# OS Installations and Upgrades in Diverse Environments

Installing and upgrading operating systems is one of the most hands-on skills a technician uses. The exam tests your ability to choose the right installation method, configure boot media, select partition schemes, pick appropriate file systems, and execute post-installation tasks correctly.

## Clean Install vs Upgrade

The first decision in any OS deployment is whether to perform a clean install or an in-place upgrade.

<ConceptCallout type="tip">When in doubt on the exam, a clean install is always the safer, more thorough answer. If the question says the machine is slow, has malware, or needs a major version change, choose clean install.</ConceptCallout>

**Supported Windows 10 upgrade paths** (in-place):
- Home → Home, Pro, Pro Education, Education
- Pro → Pro, Pro Education, Education, Enterprise
- Enterprise → Enterprise, Education

You cannot in-place upgrade from Windows 10 to Windows 11 without meeting all hardware requirements (TPM 2.0, Secure Boot, compatible CPU).

## Boot Media Types

To install an OS, the system must boot from installation media rather than the existing hard drive.

### USB Flash Drive (Most Common)

USB is the dominant installation method. It is faster than optical media, reusable, and works on modern hardware that may not have an optical drive. Tools for creating bootable USB drives include:

- **Rufus** — Windows utility, supports both MBR and GPT, UEFI and Legacy BIOS
- **Windows Media Creation Tool** — official Microsoft tool for creating Windows 10/11 USB media
- **balenaEtcher** — cross-platform, used for Linux ISOs
- **dd** (Linux/macOS command line) — low-level disk imaging to USB

<ConceptCallout type="warning">Simply copying ISO files to a USB drive does NOT make it bootable. You must use a proper tool (Rufus, Media Creation Tool) that writes the boot sector and partition structure correctly.</ConceptCallout>

### Optical Disc (Legacy)

DVD and CD installation media is now largely obsolete but may appear on the exam in legacy contexts. DVDs hold up to 8.5 GB (dual layer), which is enough for Windows installation media. Optical drives are increasingly absent from modern hardware.

### PXE / Network Boot (Enterprise)

Preboot Execution Environment (PXE) allows a computer to boot and load an OS image from the network without any local boot media. The system's NIC must support PXE, which is enabled in UEFI/BIOS settings.

**PXE boot sequence:**
1. System powers on — NIC broadcasts a DHCP discovery packet
2. DHCP server responds with an IP address and the address of a TFTP server
3. System downloads the bootloader from the TFTP server via TFTP
4. Bootloader loads the OS image or installer from the network
5. Installation or imaging proceeds without any physical media

PXE is the foundation of enterprise imaging solutions like Windows Deployment Services (WDS) and SCCM/Intune OS deployment. It allows a single technician to image hundreds of machines simultaneously across the network.

<ConceptCallout type="info">PXE requires DHCP, TFTP, and a WDS or similar server. The client NIC must be PXE-capable and the boot order must have "Network" listed before the local hard drive.</ConceptCallout>

## MBR vs GPT Partition Schemes

Before installing an OS, the drive must be partitioned. The partition scheme determines how partitions are structured on the disk.

<CompareCards
  title="Partition Scheme Comparison"
  left={{ title: "MBR (Master Boot Record)", icon: "HardDrive", color: "orange", points: ["Maximum disk size: 2 TB", "Maximum 4 primary partitions", "Uses extended + logical partitions to work around the limit", "Compatible with Legacy BIOS and older systems", "Boot code stored in first 512 bytes of disk", "No redundancy — corruption of MBR = unbootable disk"] }}
  right={{ title: "GPT (GUID Partition Table)", icon: "Shield", color: "blue", points: ["Maximum disk size: 9.4 ZB (zettabytes)", "Up to 128 primary partitions (Windows)", "Each partition has a globally unique identifier (GUID)", "Required for UEFI boot (UEFI + GPT = modern standard)", "Backup partition table stored at end of disk", "CRC32 checksums detect corruption"] }}
/>

<ConceptCallout type="warning">UEFI requires GPT. If you are doing a clean install on a modern system using UEFI mode, you must use GPT. Attempting to install in UEFI mode on an MBR disk will fail. Legacy BIOS mode can use MBR or GPT, but booting from a GPT disk in legacy mode requires special configuration.</ConceptCallout>

**Key rule for the exam:** UEFI + GPT = modern systems. Legacy BIOS + MBR = older systems. If a question mentions a disk larger than 2 TB that needs to boot, GPT is required.

## File System Comparison

The file system determines how data is organized, stored, and accessed on a partition. Choosing the wrong file system for the job causes compatibility problems.

| File System | OS | Max File Size | Max Volume | Key Features | Use Case |
|---|---|---|---|---|---|
| NTFS | Windows | 16 TB (practical) | 256 TB | Journaling, permissions, encryption (EFS), compression, ACLs | Windows system and data drives |
| FAT32 | Cross-platform | **4 GB** | 2 TB | Maximum compatibility, no permissions, no journaling | USB drives, legacy compatibility |
| exFAT | Cross-platform | 128 PB | 128 PB | Large files, no 4 GB limit, cross-platform | Large removable drives, SD cards, video files |
| ext4 | Linux | 16 TB | 1 EB | Journaling, permissions, large files, Linux-native | Linux system and data drives |
| APFS | macOS | 8 EB | 8 EB | Snapshots, clones, encryption, space sharing | macOS system drives (SSD-optimized) |

<ConceptCallout type="warning">FAT32's 4 GB file size limit is a critical exam point. If someone tries to copy a large video file (e.g., a 6 GB movie) to a USB drive formatted as FAT32, the copy will fail even if there is enough free space. Format the drive as exFAT instead.</ConceptCallout>

**NTFS key features to know:**
- **Journaling**: Logs file system transactions — allows recovery from unexpected shutdowns without data corruption
- **Permissions**: NTFS ACLs control read, write, execute access per user/group
- **EFS (Encrypting File System)**: File-level encryption tied to user account
- **Compression**: Native file/folder compression
- **Quotas**: Limit disk space per user

## Post-Installation Tasks Checklist

After the OS is installed and updated, a standard set of tasks ensures the machine is ready for use:

| Task | Why It Matters |
|---|---|
| Install all drivers | Hardware won't function properly without correct drivers |
| Run Windows Update fully | Base install has known vulnerabilities |
| Install antivirus / endpoint protection | Unprotected system is at risk immediately |
| Configure Windows Firewall | Ensure firewall rules are appropriate |
| Join domain or configure workgroup | Required before deploying to a user |
| Install required applications | Line-of-business software, productivity tools |
| Migrate user data | Transfer documents, browser favorites, email profiles |
| Verify backup is configured | Confirm the system is included in backup jobs |
| Enable BitLocker (if Pro/Enterprise) | Encrypt the drive before handing to user |
| Test application functionality | Confirm all required apps launch and work |

## Upgrade Paths and the Windows Upgrade Assistant

For in-place upgrades, Microsoft provides the **PC Health Check** tool (for Windows 11 readiness) and the **Windows Upgrade Assistant** (for upgrading from Windows 10 to Windows 11).

**Before an in-place upgrade:**
1. Run the compatibility check tool
2. Verify all hardware drivers have Windows 11 versions available
3. Back up the system completely (non-negotiable)
4. Disconnect non-essential peripherals
5. Ensure at least 20 GB free disk space
6. Plug in the laptop (never upgrade on battery)

**After an in-place upgrade:** Windows creates a `Windows.old` folder containing the previous installation. This allows rollback via Settings > Recovery > Go back within 10 days. After 10 days, the option disappears and Windows.old can be deleted via Disk Cleanup to reclaim space.

## Unattended and Automated Installations

In enterprise environments, manually clicking through the installation wizard for every machine is impractical. Automated deployment methods include:

**Answer Files (unattend.xml):** An XML file that pre-populates installation wizard responses — language, product key, partition layout, regional settings, user account creation. Created using the Windows System Image Manager (SIM) tool, part of the Windows Assessment and Deployment Kit (ADK). The answer file is placed on the USB drive or a network share and referenced during installation.

**Windows Deployment Services (WDS):** A Windows Server role that enables PXE network boot and OS deployment across the network. WDS hosts WIM (Windows Imaging Format) files and answer files. Technicians can deploy a standard image to many machines simultaneously without touching each one physically.

## Exam Scenario Quick Reference

| Scenario | Correct Choice |
|---|---|
| New hire, fresh hardware, never had an OS | Clean install |
| Customer wants Windows 11, currently on Windows 10, hardware compatible | In-place upgrade |
| Machine has severe malware infection | Clean install |
| 50 machines need the same OS + apps deployed overnight | WDS + PXE imaging |
| USB drive won't accept a 5 GB file | Reformat as exFAT (FAT32 has 4 GB file limit) |
| New 3 TB hard drive won't show more than 2 TB | Convert to GPT |
| UEFI system won't boot Windows from disk | Ensure disk is GPT, not MBR |
| Customer needs cross-platform USB compatibility | FAT32 (small files) or exFAT (large files) |
